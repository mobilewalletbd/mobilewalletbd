  🔴 ERROR 2: TEAMS FIRESTORE PERMISSION DENIED (CRITICAL)
================================================================================

ERROR:
  W/Firestore: Listen for Query(target=Query(teams where memberIds
  array_contains Z1vuF6tM9oQdWN2VV2agpxk5vpJ3 order by __name__))
  failed: Status{code=PERMISSION_DENIED, description=Missing or
  insufficient permissions.}

DESCRIPTION:
  The Dart code queries teams using the "memberIds" field:
    .where('memberIds', arrayContains: userId)

  But the Firestore rules check the "members" field:
    request.auth.uid in resource.data.members

  This is a FIELD NAME MISMATCH between code and rules:
    - Code queries: "memberIds" (array of user ID strings)
    - Rules check:  "members"   (which is actually an array of objects)

  Additionally, Firestore Security Rules cannot evaluate list queries
  (arrayContains) on read rules. For list queries, the rule must check
  a field that the query itself filters on, or use a simpler approach.

SOLUTION:
  Update firestore.rules to check "memberIds" instead of "members":

  BEFORE (current - broken):
    allow read: if isAuthenticated() &&
                   (resource.data.ownerId == request.auth.uid ||
                    request.auth.uid in resource.data.members);

  AFTER (fixed):
    allow read: if isAuthenticated() &&
                   (resource.data.ownerId == request.auth.uid ||
                    request.auth.uid in resource.data.memberIds);

  Also fix the update rule:
  BEFORE (current - broken):
    allow update: if isAuthenticated() &&
                     (resource.data.ownerId == request.auth.uid ||
                      resource.data.members[request.auth.uid] == 'admin');

  AFTER (fixed - use memberIds for membership, check members for role):
    allow update: if isAuthenticated() &&
                     (resource.data.ownerId == request.auth.uid ||
                      request.auth.uid in resource.data.memberIds);

  NOTE: The admin role check (members[uid] == 'admin') won't work because
  "members" is an array of objects, not a map. If you need admin-only
  updates, restructure "members" as a map: { "uid1": "admin", "uid2": "member" }
  OR add a separate "memberRoles" map field.

  File to modify:
    g:\wallet\mobilewalletbd\firestore.rules (lines 179-196)

ERROR:

  E/GoogleApiManager: java.lang.SecurityException: Unknown calling
  package name 'com.google.android.gms'.
  W/GoogleApiManager: ConnectionResult{statusCode=DEVELOPER_ERROR}

DESCRIPTION:
  This error means Google Play Services cannot verify your app's identity.
  The most common cause is a SHA-1 certificate fingerprint mismatch:
    - The app is signed with a debug key that is NOT registered in the
      Firebase Console for the Android app "com.mobilewallet.v1".
    - The google-services.json is missing the SHA-1 based oauth_client.

  Impact:
    - Google Sign-In will fail completely.
    - ProviderInstaller cannot update (security provider outdated).
    - Some Firebase features may not work correctly.

  Note: This error is MUCH more severe on physical devices. On emulators
  it may partially work because Firebase Auth uses a different path.

SOLUTION:
  1. Get your current debug SHA-1 fingerprint:
     Run in terminal (from android/ folder):
       gradlew.bat signingReport

     Look for the "debug" variant > SHA1 line.

  2. Add SHA-1 to Firebase Console:
     a. Go to https://console.firebase.google.com/
     b. Select project "smartwallet-c97fe"
     c. Go to Project Settings > General
     d. Scroll to "Your apps" > Android app (com.mobilewallet.v1)
     e. Click "Add fingerprint"
     f. Paste your debug SHA-1 and save

  3. Download updated google-services.json:
     a. Click "Download google-services.json" button
     b. Replace: g:\wallet\mobilewalletbd\android\app\google-services.json
     c. The new file will have oauth_client entries with client_type: 1
        (Android client) which are currently missing.

  4. Rebuild:
     flutter clean && flutter run

  E/FlutterJNI: android.graphics.ImageDecoder$DecodeException:
  Failed to create image decoder with message 'unimplemented'
  Input contained an error.

DESCRIPTION:
  Flutter's Impeller rendering backend failed to decode an image asset.
  This can happen when:
    a) An image file is corrupted or in an unsupported format (e.g., WebP
       with animation, or a malformed PNG/JPEG).
    b) The image is being loaded from a network URL that returned an error
       page (HTML) instead of an actual image.
    c) The emulator's GPU doesn't support certain image codecs.

  This error appeared right after the sign-out event, suggesting it may
  be related to loading a user profile photo or avatar placeholder.

SOLUTION:
  a) Check all image assets in assets/ folder for corruption:
     - Try opening each image in a normal image viewer.
     - Ensure no SVG files are being loaded as raster images.

  b) Add error handling for network images:
     Use Image.network() with an errorBuilder:
       Image.network(
         url,
         errorBuilder: (context, error, stackTrace) =>
           Icon(Icons.person, size: 48),
       )

  c) If the issue is with a specific widget, wrap the image loading with
     a try-catch or use a fallback icon/placeholder.

  d) If this only happens on the emulator, it may be an Impeller issue.
     Test on a physical device to confirm.

  🟠 ERROR 5: AUTH STATE FLICKERING - SIGN OUT THEN SIGN IN (MEDIUM)
================================================================================

ERROR:
  D/FirebaseAuth: Notifying auth state listeners about a sign-out event.
  ... (GC pause) ...
  D/FirebaseAuth: Notifying auth state listeners about user (Z1vuF6...vpJ3).

DESCRIPTION:
  Firebase Auth fires a sign-out event followed immediately by a sign-in
  event for the same user. This "auth state flicker" can cause:
    - Brief flash of the login screen.
    - Streams/listeners to reset and re-fetch data unnecessarily.
    - The session clear we see: "Session cleared" followed by "Token refreshed".

  This happens because Firebase Auth re-authenticates after a token refresh,
  which temporarily appears as a sign-out.

SOLUTION:
  In auth_provider.dart, debounce the auth state changes:
    - Add a short delay (500ms) before reacting to sign-out events.
    - Or use authStateChanges().distinct() to filter duplicate states.
    - Or switch to idTokenChanges() which is more granular.

  Example:
    _firebaseAuth.authStateChanges()
      .debounceTime(Duration(milliseconds: 500))
      .distinct((a, b) => a?.uid == b?.uid)
      .listen((user) { ... });
