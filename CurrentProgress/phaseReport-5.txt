BUG-01 [CRITICAL] — TabController created without TickerProviderStateMixin
  File: lib/features/digital_card/presentation/screens/card_editor_screen.dart
  Line: 56
  Issue: TabBar uses:
           TabController(length: 3, vsync: Scaffold.of(context))
         TabController requires a TickerProvider (vsync). Using Scaffold.of()
         as vsync is INCORRECT. Flutter will throw:
           "Improper use of a BuildContext" or "vsync must be a TickerProvider"
  Fix Required:
    1. Add `with SingleTickerProviderStateMixin` to _CardEditorScreenState
    2. Create late final TabController _tabController in initState()
    3. Dispose _tabController in dispose()
    4. Reference _tabController instead of creating inline
  Status: ❌ BUG — WILL CRASH ON OPEN

BUG-02 [HIGH] — JPG export does not actually apply JPG format
  File: lib/features/digital_card/data/services/card_image_export_service.dart
  Line: 38
  Issue: When `asPng = false` (JPG export), the code still uses:
           format: asPng ? ui.ImageByteFormat.png : ui.ImageByteFormat.png
         Both branches are `.png`. JPG is never used.
         The file extension will say `.jpg` but the bytes are PNG.
         Some apps/parsers will reject the mismatch.
  Fix Required:
           format: asPng ? ui.ImageByteFormat.png : ui.ImageByteFormat.rawRgba
         (Note: Flutter's dart:ui has no native JPEG encoder — rawRgba → 
          use `image` package to compress to JPEG bytes, or always export PNG.)
  Status: ❌ BUG — JPG EXPORT SILENTLY PRODUCES PNG

BUG-03 [HIGH] — QrPainter.withQr `foregroundColor` param is deprecated
  File: lib/features/digital_card/data/services/qr_code_service.dart
  Line: 98–104
  Issue: QrPainter.withQr(qr:..., color: foregroundColor, ...)
         In qr_flutter >=4.0: `color` was renamed to `dataModuleStyle` +
         `eyeStyle`. Using the old `color:` named param may still compile
         against old API but can produce lint warnings or fail on package
         upgrade. The `foregroundColor` named param in `generateQrImage()`
         is also the old naming convention.
  Fix Required:
    Replace with:
      QrPainter.withQr(
        qr: qrCode!,
        dataModuleStyle: QrDataModuleStyle(
          dataModuleShape: QrDataModuleShape.square,
          color: foregroundColor,
        ),
        eyeStyle: QrEyeStyle(eyeShape: QrEyeShape.square, color: foregroundColor),
        gapless: true,
      )
  Status: ⚠️ DEPRECATION — WILL WARN, MAY BREAK ON UPGRADE

BUG-04 [MEDIUM] — Duplicate @override annotation in repository impl
  File: lib/features/digital_card/data/repositories/card_design_repository_impl.dart
  Lines: 148-149 and 175-176
  Issue: Two consecutive `@override` annotations before each of:
           getVersionHistory() (line 148-150)
           restoreVersion() (line 175-177)
         Double @override causes an analysis warning:
           "The annotation @override was probably already added"
  Fix Required: Remove the duplicate @override annotation from each method.
  Status: ⚠️ LINT WARNING — CAUSES ANALYZER NOISE

BUG-05 [MEDIUM] — themeColor hex string encodes ARGB, not RRGGBB, in provider
  File: lib/features/digital_card/presentation/providers/card_design_provider.dart
  Line: 176
  Issue: When a template is selected, the theme color is set via:
           '#${template.primaryColor.toARGB32().toRadixString(16).padLeft(8, '0')}'
         This produces an 8-character hex like `#FF0BBF7D` (with alpha prefix),
         but the PDF service and QR service's `_hexToColor()` methods only
         handle 6 or 7 character hex strings. An 8-char ARGB hex will parse
         incorrectly → wrong color in PDF/QR.
  Fix Required: Strip the alpha channel:
           '#${(template.primaryColor.toARGB32() & 0xFFFFFF).toRadixString(16).padLeft(6, '0')}'
  Status: ❌ BUG — WRONG COLOR IN PDF/QR WHEN TEMPLATE SELECTED
