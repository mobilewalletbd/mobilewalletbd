2.1 TECHNOLOGY STACK
--------------------
- Framework: Flutter (Latest Stable)
- Language: Dart
- State Management: **Riverpod** (Modern, robust, testable).
- Local Database (Offline-First): **Isar** (High performance, ACID compliant, NoSQL).
  * Why: Ensures full app functionality even without internet.
- Backend / Cloud (V1): **Microsoft Azure** (Student Encrypted Plan).
  * **Auth**: Azure Active Directory B2C (Enterprise-grade security, Free Tier).
  * **Database**: Azure Cosmos DB for NoSQL (Permanent Free Tier).
  * **Storage**: Azure Blob Storage (Uses Student Credit).
  * **API**: Azure Functions (Serverless, Event-driven).
- Backup & Data Portability:
  * **Unified Cloud Sync**: All data automatically synced to Azure Cosmos/Blob.
  * **Manual Export**: Export data as JSON/CSV/PDF for user control.
- Navigation: GoRouter.

2.1.1 AZURE STORAGE STRATEGY (Student Plan Optimized)
------------------------------------------------------
1. Local Storage (Primary - Free & Fast): 
   - High-res card images and data stored in device's local App Directory (Isar).
   - Ensures instant loading and offline capability.
2. Cloud Sync (Azure Ecosystem):
   - **Data**: Synced to **Azure Cosmos DB** (NoSQL Container).
   - **Images**: Uploaded to **Azure Blob Storage** (Hot Tier).
   - **Optimization**: Use 'Hot/Cool' storage tiers to manage credit usage efficiently.
   - **Authentication**: Azure AD B2C handles all token management and security access.

2.2 ARCHITECTURAL PATTERN: FEATURE-FIRST + RIVERPOD
-----------------------------------------------------
We follow a strict **Feature-First Layered Architecture**. Code is organized by "what it does" (feature) rather than "what it is" (layer).

### A. Directory Structure
/lib
  /src
    /app.dart                # Main App Widget (Providers/Theme/Routing)
    /features                # FEATURE MODULES
      /auth
        /data                # Repositories, DTOs, Data Sources
        /domain              # Entities, Failures, Value Objects
        /presentation        # Screens, Widgets, Controllers (Riverpod)
      /contacts
      /wallet
      /settings
      /shared                # Shared between features (e.g. Card Scanner)
    /core                    # CORE INFRASTRUCTURE
      /config                # Environment variables, Constants
      /error                 # Global Failure classes, Exceptions
      /theme                 # AppTheme, Colors, Typography
      /router                # GoRouter configuration
      /utils                 # Formatters, Validators (Currency, Date)
    /common_widgets          # Global reusable widgets (Buttons, Inputs)

### B. Layer Responsibilities
1. **Domain Layer (The Core)**
   - PURE DART. No Flutter dependencies (ideally).
   - Contains: Entities (Models), Value Objects.
   - Purpose: Defines *what* the business data is.
   
2. **Data Layer (The Implementation)**
   - Depends on Domain.
   - Contains: Repositories (Impl), Data Sources (API/DB clients), DTOs (JSON parsers).
   - Purpose: Fetches data from Isar/Supabase and converts to Domain Entities.
   - *Riverpod Providers*: `authRepositoryProvider`, `contactsRepositoryProvider`.

3. **Presentation Layer (The UI)**
   - Depends on Domain & Application.
   - Contains: Screens, Widgets, Controllers (AsyncNotifier/StateNotifier).
   - Purpose: Displays state and captures user events.
   - *Riverpod Providers*: `authControllerProvider` (manages UI state).

### C. Data Flow
1. **User Interaction**: User taps "Save Contact".
2. **Controller**: `ContactController` receives event.
3. **Repository**: Controller calls `ContactRepository.addContact()`.
4. **Data Source**: Repository calls `IsarDataSource.write()`.
5. **State Update**: Controller updates state (Loading -> Success).
6. **UI Rebuild**: UI listens to Controller and rebuilds to show "Saved".

3. GlobalContact (Source of Truth)

4. CardDesign (My Card)
   - Core: { id, contact_id, logo_url, theme_color, layout_type, custom_qr_style }

5. Wallet (Internal Ledger)

5. Team (V1 Basic)
   - Core: { id, name, owner_id, created_at }
   - Membership: { members[]: { user_id, role (ADMIN/MEMBER), joined_at } }
   - Stats: { shared_contacts_count }

2.4 SYNC & OFFLINE STRATEGY (CRITICAL)
--------------------------------------
A. Offline-First Principles
   - **Write**: All writes go to Isar (Local DB) first. UI updates immediately (Optimistic).
   - **Queue**: Background job queues the operation for sync.
   - **Sync**: When online, queue is processed and sent to Supabase.

B. Strategies
   - **Up-Sync**: Pushing local changes (New Contact, Edit) to Cloud.
   - **Down-Sync**: Fetching updates (Profile changes, Team data) from Cloud.
C. Global Sync Logic (Azure Functions)
   - **Trigger**: Scan/Edit uploads to queue -> Azure Function triggers.
   - **Compare**: Function checks `CosmosDB.GlobalContacts` vs `UserContact.version`.
   - **Prompt UI**: "Newer info found" notification (via Azure Notification Hubs).
   - **Privacy**: User's private notes stored in Cosmos private partition (`/userId`).
   - **Override**: Owner-verified data always wins.

3. SECURITY & PERFORMANCE SPECIFICATIONS
================================================================================

3.1 SECURITY SPECIFICATIONS (V1 PRIORITY)
-----------------------------------------
A. Authentication & Session
   - **Tokens**: Short-lived JWT Identity Tokens + Rotating Refresh Tokens (Supabase).
   - **MFA**: OTP via SMS for login/registration (prevent brute force).
   - **Session**: Device binding (track Device ID). Auto-logout on token expiry.

B. Data Encryption & Privacy
   - **At Rest**: Isar Database encryption (AES-256) for local sensitive data.
   - **In Transit**: All API calls over TLS 1.2+ (HTTPS).
   - **Backups**: Google Drive uploads are user-encrypted.
   - **Privacy**: Masking sensitive fields (e.g., +880 17**-***456) in UI.

C. Access Control (RBAC)
   - **User Role**: Standard access to own contacts.
   - **Team Admin**: Permission to add/remove members (future prep).

D. API & RLS POLICIES (SUPABASE)
--------------------------------
   - **Principle**: No Backend Middleware. Security is enforced at the Database Layer.
   - **Policy 1 (Private)**: `auth.uid() = user_id` for Contacts/Wallets.
   - **Policy 2 (Shared)**: `auth.uid() IN (select member_id from team_members)` for Team Directories.
   - **Edge Functions**: Used only for complex logic (e.g., inviting a user to a team).

3.2 PERFORMANCE OPTIMIZATION STRATEGY (DETAILED)
----------------------------------------------
I. PERFORMANCE GOALS
   - **Startup**: < 1.5s (Cold Start) on mid-range devices.
   - **Responsiveness**: 60 FPS UI rendering; < 300ms API latency.
   - **Bundle Size**: < 25MB initial download (Split APKs).

II. CLIENT-SIDE OPTIMIZATION
   - **App Bundle**: Use Split APKs (per architecture) + Code Shrinking (Tree-shaking).
   - **Modular Loading**: Load features lazily (e.g., Scan library only when needed).
   - **Lists**: Use `ListView.builder` with `const` widgets and virtualization.
   - **State Management**: Use Riverpod `select()` to avoid rebuilding entire widget trees.
   - **Background Work**: Isolate critical tasks (OCR, large JSON parsing) from UI thread.

III. IMAGE & ASSET COMPRESSION (CRITICAL)
   - **Capture Resolution**: Crop to card ratio before processing (don't use full sensor).
   - **Compression**: WebP format (Quality ~75%) on device before upload.
   - **Thumbnails**: Server generated derivatives; Load low-res first.
   - **Caching**: `cached_network_image` with long TTL.
   - **Assets**: Prefer SVGs over PNGs.

IV. CACHING STRATEGY
   - **Local DB (Isar)**: Acts as L1 Cache. All reads hit Isar first.
   - **Memory**: Cache frequently accessed user profiles & recent transactions (Riverpod KeepAlive).
   - **Network**: Standard HTTP Caching headers (ETag) for static resources.

V. MEMORY MANAGEMENT
   - **Lists**: Pagination (Infinite Scroll) to limit items in memory.
   - **Images**: Aggressive eviction of large image caches on low-memory warnings.
   - **Pooling**: Reuse heavy objects where possible.

VI. DATABASE & QUERY (SUPABASE/ISAR)
   - **Indexing**: Composite indexes on (user_id + created_at) for fast sorting.
   - **Pagination**: STRICT pagination on all lists (limit: 20/50).
   - **Projections**: Fetch only needed columns (e.g., list view needs name/avatar, not full bio).

VII. UX PERFORMANCE
   - **Perceived Speed**: Skeleton loaders (Shimmer) vs Spinner.
   - **Optimistic UI**: Update UI immediately on "Save/Like", rollback on error.
   - **Feedback**: Instant tap feedback (Ripple/Toast).
   - **Low Data Mode**: Auto-disable sync/high-res images on weak connections.

VIII. LOAD TESTING
    - **Scenario**: 1000 concurrent "Add Contact" ops to verify sync stability.
    - **Device Test**: Verify 60FPS on lowest supported Android device (e.g., 2GB RAM).
