================================================================================
OVERVIEW: PHASE-BASED DEVELOPMENT STRATEGY
================================================================================

The Smart Contact Wallet V1 project has been divided into 7 major phases,
each representing a logical development milestone. This phased approach:

✓ Enables parallel team work where possible
✓ Provides clear checkpoint deliverables
✓ Maintains dependency order (can't build UI without backend)
✓ Allows for iterative testing at each phase
✓ Simplifies project tracking and progress reporting

CRITICAL PATH: Phases 1 → 2 → 3 must be completed sequentially
PARALLEL WORK: Phases 4-6 can be developed simultaneously after Phase 3
FINAL PHASE: Phase 7 integrates everything and prepares for deployment

================================================================================
PHASE 1: FOUNDATION & INFRASTRUCTURE SETUP
================================================================================
Duration: 1-2 Weeks
Team Members: 1-2 Backend/DevOps Developers
Dependencies: None (Starting Phase)
Completion Criteria: Firebase services operational, Flutter project initialized

OBJECTIVE:
----------
Establish the complete technical infrastructure including Firebase cloud services,
project scaffolding, configuration management, and core service integrations.
No UI work in this phase - pure backend and setup.

DELIVERABLES:
-------------
1.1 Firebase Cloud Infrastructure
   - Firebase Project created and configured
   - Firebase Authentication enabled (Email, Phone, Google Sign-In)
   - Cloud Firestore initialized with security rules
   - Cloudinary account configured for image storage
   - Android/iOS apps registered in Firebase Console

1.2 Flutter Project Initialization
   - Project created with proper package structure
   - All dependencies installed (Riverpod, Isar, Firebase packages)
   - Analysis options configured with strict linting
   - Build runner executed successfully
   - Git repository initialized with proper .gitignore

1.3 Configuration & Security
   - .env file template created with all required variables
   - assets/config/required_info.json populated
   - Environment variable loading system implemented
   - AppConfig class with validation logic
   - Firebase Security Rules implemented

1.4 Core Services
   - FirebaseAuthService (Authentication integration)
   - FirestoreService (Cloud database integration)
   - CloudinaryService (Image upload service)
   - IsarService (local database initialization)
   - AppRouter (GoRouter configuration with routes)
   - Error handling infrastructure
   - Logging system setup

1.5 Documentation
   - ENV_details.txt guide for developers
   - FIREBASE_SYSTEM_ARCHITECTURE.txt for reproducible setup
   - Developer onboarding README
   - API documentation structure

TESTING CHECKPOINTS:
--------------------
✓ Firebase Authentication user flow test passes
✓ Firestore test document can be created and queried
✓ Cloudinary test image can be uploaded
✓ Flutter app launches without crashing
✓ AppConfig loads without errors
✓ Offline-first sync works correctly

PHASE 2: AUTHENTICATION & USER MANAGEMENT
================================================================================
Duration: 1-2 Weeks
Team Members: 1-2 Flutter Developers
Dependencies: Phase 1 (Azure AD B2C must be configured)
Completion Criteria: Users can sign up, log in, and maintain sessions

OBJECTIVE:
----------
Implement complete authentication flow using Azure AD B2C, including user
registration, login, OTP verification, session management, and local caching.

DELIVERABLES:
-------------
2.1 Domain Layer (Pure Business Logic)
   - AuthUser entity with validation
   - AuthFailure sealed class hierarchy (7 failure types)
   - AuthRepository interface (contract)

2.2 Data Layer (Implementation)
   - AuthRemoteDataSource (Azure AD B2C integration)
   - AuthLocalDataSource (Isar cache for offline)
   - AuthRepositoryImpl with Either<Failure, Success> pattern
   - Local Isar schema: AuthUserLocal collection

2.3 Presentation Layer (UI Screens)
   - Splash Screen (loading animation)
   - Welcome/Logo Page (brand introduction)
   - Login Screen (email/phone + password)
   - Register Screen (multi-field form with validation)
   - Forgot Password Screen (password reset flow)
   - OTP Verification Screen (6-digit input boxes)

2.4 State Management
   - AuthController (Riverpod AsyncNotifier)
   - Auth state providers (isAuthenticated, currentUser)
   - Session persistence logic
   - Auto-logout on token expiry

2.5 Navigation Guards
   - Router redirect logic (unauthenticated → login)
   - Deep linking for OAuth callback
   - Post-auth navigation to home

TESTING CHECKPOINTS:
--------------------
✓ New user can register with valid email
✓ Existing user can log in successfully
✓ Invalid credentials show proper error messages
✓ OTP code verification works (if enabled)
✓ "Forgot Password" flow sends reset email
✓ User session persists after app restart (offline mode)
✓ Token refresh happens automatically before expiry

KEY USER FLOWS TO TEST:
------------------------
1. Fresh Install → Welcome → Sign Up → OTP → Home
2. Existing User → Splash → Auto-Login → Home
3. Logged Out User → Welcome → Login → Home
4. Password Reset → Forgot Password → Email → Reset → Login

PHASE 3: CONTACT MANAGEMENT CORE
================================================================================
Duration: 2-3 Weeks
Team Members: 2-3 Flutter Developers
Dependencies: Phase 2 (Authentication must work for user context)
Completion Criteria: Users can add, view, edit, and delete contacts

OBJECTIVE:
----------
Implement the heart of the application - contact management with CRUD operations,
local-first storage, cloud sync, and basic organization features.

DELIVERABLES:
-------------
3.1 Domain Layer
   - Contact entity (full data model)
   - Category & Tag entities
   - ContactRepository interface
   - Failure types (NetworkFailure, ValidationFailure, etc.)

3.2 Data Layer - Local (Isar)
   - Contact Isar collection schema (@collection annotated)
   - Indexes: userId, category, isFavorite, createdAt
   - ContactLocalDataSource with CRUD methods
   - Query methods (search, filter, pagination)

3.3 Data Layer - Remote (Azure Cosmos DB)
   - ContactRemoteDataSource (HTTP calls to Azure Functions)
   - Sync logic (upload local contacts to cloud)
   - Conflict resolution strategy
   - Background sync queue

3.4 Presentation Layer - Screens
   - Home Dashboard (contact grid/list view)
   - Contact List View (with search bar)
   - Add Contact Method Selection (Scan / Manual / Import)
   - Manual Contact Form (all fields with validation)
   - Contact Details Page (full profile view)
   - Edit Contact Screen (pre-filled form)
   - Contact Full-Screen Card View (image zoom)

3.5 Presentation Layer - Widgets
   - ContactCard (grid item with image + name)
   - ContactListTile (list row format)
   - CategoryChip (pill-shaped category selector)
   - SearchBar with real-time filtering
   - EmptyContactsWidget (when no contacts exist)

3.6 State Management
   - ContactsController (manages contact list state)
   - ContactDetailController (single contact state)
   - Providers: contactsProvider, categoriesProvider, favoritesProvider

3.7 Features
   - Add contact (manual entry only in this phase)
   - Edit contact
   - Delete contact (with confirmation dialog)
   - Mark as favorite (star icon toggle)
   - Assign category (Business, Personal, etc.)
   - Global search (name, company, phone, email)
   - Filter by category
   - Sort options (A-Z, recently added, recently contacted)

TESTING CHECKPOINTS:
--------------------
✓ Add new contact via manual form saves to Isar
✓ Contact appears in list immediately
✓ Edit contact updates data correctly
✓ Delete contact removes from list
✓ Search finds contacts by partial name/company
✓ Filter by category shows only relevant contacts
✓ Favorite toggle works and persists
✓ Pagination loads 20 contacts at a time
✓ Contact syncs to Cosmos DB when online

BLOCKING RISKS:
---------------
⚠ Isar schema changes require app reinstall during development
⚠ Large contact lists (1000+) may cause performance issues → Optimize queries

PHASE 4: BUSINESS CARD SCANNING & OCR
================================================================================
Duration: 2 Weeks
Team Members: 1-2 Flutter Developers (can run parallel to Phase 5)
Dependencies: Phase 3 (Contacts module must exist to save scanned data)
Completion Criteria: Users can scan business cards and extract information

OBJECTIVE:
----------
Implement camera-based business card scanning with OCR (Optical Character
Recognition) to automatically extract contact information from card images.

DELIVERABLES:
-------------
4.1 Camera Integration
   - Camera permissions handling (Android/iOS)
   - Custom camera screen with alignment guide
   - Capture front and back of card
   - Image cropping to card aspect ratio
   - Flash toggle for low-light conditions

4.2 OCR Processing
   - Integration with Microsoft Azure Computer Vision API (OCR)
   - OR Google ML Kit Text Recognition (free alternative)
   - Text extraction from card image
   - Field mapping logic (Name, Title, Company, Phone, Email, Address)
   - Confidence scoring for extracted fields

4.3 Presentation Layer
   - Scan Card Screen (camera viewfinder with guide overlay)
   - OCR Preview Screen (shows extracted data with editable fields)
   - Low-confidence field highlighting (yellow background)
   - Manual correction option before saving

4.4 Data Processing
   - Phone number formatting (+880 for Bangladesh)
   - Email validation
   - Company name standardization
   - Duplicate detection (compare with existing contacts)

4.5 Image Storage
   - Save card images locally (Isar embedded)
   - Upload to Azure Blob Storage (card-images container)
   - Thumbnail generation for list views
   - Image compression (WebP format, 75% quality)

TESTING CHECKPOINTS:
--------------------
✓ Camera permission prompt appears on first use
✓ Alignment guide helps user position card correctly
✓ Image captured is clear and properly cropped
✓ OCR extracts at least 70% of fields correctly
✓ User can edit OCR results before saving
✓ Scanned contact saves to database
✓ Card images stored in Blob Storage
✓ Duplicate warning shown if contact already exists

PERFORMANCE TARGETS:
--------------------
- OCR processing time: < 3 seconds
- Image upload time: < 5 seconds on 4G
- Card image size: < 500KB after compression

PHASE 6: INTERNAL WALLET (BOOKKEEPING ONLY)
================================================================================
Duration: 1-2 Weeks
Team Members: 1-2 Flutter Developers
Dependencies: Phase 3 (Needs contacts to send/receive from)
Completion Criteria: Users can simulate wallet transactions (no real money)

OBJECTIVE:
----------
Implement internal wallet ledger system for bookkeeping demonstration. This
prepares the UI/UX for V2 payment integration but does NOT move real money.

DELIVERABLES:
-------------
6.1 Domain Layer
   - Wallet entity (balance, currency: BDT)
   - WalletTransaction entity (type, amount, counterparty, timestamp)
   - Transaction types: CREDIT, DEBIT, TRANSFER
   - TransactionStatus: PENDING, COMPLETED, FAILED

6.2 Data Layer
   - Wallet Isar collection
   - WalletTransaction Isar collection
   - Indexes: userId, walletId, timestamp
   - Mock transaction processing logic

6.3 Presentation Layer - Screens
   - Wallet Home (balance card + recent transactions)
   - Add Money Screen (mock demo only)
   - Send Money Screen (select contact → enter amount → confirm)
   - Receive Money Screen (show QR code with wallet ID)
   - Transaction History (filterable list)
   - Transaction Detail Screen (full transaction view)

6.4 Features
   - View wallet balance (starts at ৳0)
   - Add Money (demo: adds to balance instantly)
   - Send Money to Contact (deducts from sender, logs transaction)
   - Transaction history with filters (Sent/Received, Date range)
   - Search transactions by contact name
   - Contextual "Pay" button on Contact Detail page

6.5 UI Components
   - Virtual wallet card (credit card design)
   - Transaction list item (icon, name, amount, date)
   - Balance display with currency symbol (৳)
   - Amount input with number keyboard
   - Confirmation dialog for transactions

TESTING CHECKPOINTS:
--------------------
✓ Initial balance is ৳0
✓ "Add Money" increases balance
✓ "Send Money" decreases sender balance
✓ Transaction appears in history
✓ Filter by date range works
✓ Search finds transactions by contact
✓ Negative balance prevented (validation error)
✓ Transaction detail shows all info

IMPORTANT NOTES:
----------------
⚠ This is SIMULATION ONLY - no real payment APIs
⚠ All transactions stored locally in Isar
⚠ Balance is just a number in database, not connected to bank
⚠ V2 will integrate bKash, Nagad, Rocket APIs

CRITICAL SUCCESS FACTORS
================================================================================

1. **Azure Infrastructure Stability**
   - Monitor costs weekly (target: < $10/month during development)
   - Set up budget alerts in Azure Portal
   - Keep backups of Cosmos DB container data

2. **Local-First Architecture**
   - Never block UI on network calls
   - All writes go to Isar first, sync in background
   - Graceful handling of offline mode

3. **Code Quality Gates**
   - All PRs require code review before merge
   - Run `flutter analyze` with zero warnings before commits
   - Fix all lint errors immediately
   - Write tests for critical business logic (80%+ coverage goal)

4. **Communication & Documentation**
   - Daily standups (15 minutes)
   - Weekly sprint planning
   - Update task.md and phase tracker after every feature completion
   - Document all API endpoints and data schemas

5. **User Feedback Loop**
   - Internal testing after each phase
   - Collect feedback from 5-10 beta users
   - Iterate based on usability issues
   - Track all bugs in GitHub Issues

RISK MITIGATION STRATEGIES
================================================================================

RISK 1: Azure Student Account Expires After 1 Year
MITIGATION:
- Cosmos DB free tier is PERMANENT (doesn't expire)
- Blob Storage cost is minimal ($2-5/month) - can pay from pocket
- Plan revenue model (V2 premium features) before Year 1 ends

RISK 2: OCR Accuracy Issues (Phase 4)
MITIGATION:
- Test with 50+ real business cards early
- Allow manual correction of all OCR fields
- Consider switching to Google ML Kit if Azure CV underperforms
- Target 70% accuracy, users can fix rest manually

RISK 3: Performance Degradation with Large Data Sets
MITIGATION:
- Test with 1000+ contacts early in Phase 3
- Implement pagination everywhere
- Use Isar indexes correctly
- Profile app with Flutter DevTools regularly

RISK 4: Team Member Availability
MITIGATION:
- Document all code with clear comments
- Use feature branches and pull requests
- Keep comprehensive README for onboarding
- Record Loom videos for complex features

RISK 5: Scope Creep (Adding Features Not in V1 Plan)
MITIGATION:
- Strict adherence to MASTER_PROJECT_PLAN_V1.txt
- All feature requests go to "V2 Backlog"
- Weekly scope review in standup
- Product owner has veto power on scope changes
